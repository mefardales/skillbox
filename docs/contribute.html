Contribute a Skill

Have deep expertise in a technology? Share it with thousands of developers. Skills are Markdown files — simple to write, high impact to contribute.

On this page
Create a Skill Writing Guide Quality Standards Submit a PR

Create a Skill

Creating a skill takes about 30–60 minutes. The entire skill is a single SKILL.md Markdown file. Here is the full process from start to finish.

1
Fork and clone the repository

Start by forking the Skillbox repo on GitHub and cloning your fork locally.

Clone your fork 
git clone https://github.com/YOUR_USERNAME/skillbox.git
cd skillbox
npm install
2
Create the skill directory

Create a new directory under skills/<category>/<skill-name>/. The skill name must be in kebab-case and match an appropriate category.

Directory structure 
mkdir -p skills/backend/my-new-skill
touch skills/backend/my-new-skill/SKILL.md
3
Write the SKILL.md file

Open SKILL.md and add the YAML frontmatter followed by your skill content. See the Skill Format docs for the required fields.

SKILL.md template 
---
name: my-new-skill
description: A concise one-sentence description of what this skill teaches the AI.
category: backend
tags: [tag1, tag2, tag3, tag4]
version: 1.0.0
authors: [your-github-username]
---

# My New Skill

## When to Use This Skill

Describe the scenarios where this skill applies.

## Core Patterns

### Pattern Name

Explain the pattern with a code example.

## What to Avoid

- Anti-pattern 1 — why it is problematic
- Anti-pattern 2 — what to do instead
4
Run the validator

Skillbox includes a validation script that checks your skill for required fields, format issues, and common mistakes.

Validate your skill 
npx tsx scripts/validate-skills.ts
5
Open a pull request

Push your branch and open a pull request against main. The PR template will guide you through the review checklist.

Push and open PR 
git checkout -b skill/my-new-skill
git add skills/backend/my-new-skill/SKILL.md
git commit -m 'feat: add my-new-skill to backend category'
git push origin skill/my-new-skill

Writing Guide

The most common mistake when writing a skill is treating it like documentation. Skills are instructions for an AI, not a learning resource for humans. Keep this distinction in mind throughout.

Write Instructions, Not Explanations

The AI already knows what React is. Do not explain React — tell the AI how you want React code written in your organization or for this project.

Good vs. poor skill content 
# Poor — explains concepts (the AI already knows this)
React hooks are functions that let you use state and lifecycle
features in functional components. useState returns an array...

# Good — tells the AI what to do
Always use the useCallback hook for event handlers passed to
child components to prevent unnecessary re-renders. Prefer
useMemo only when measuring an actual performance problem.

Include Concrete Code Examples

Every pattern should have a working code example. The AI learns from examples more than prose descriptions.

Be Explicit About Preferences

Do not hedge. Say "always use X" and "never use Y" rather than "consider using X" or "Y might cause problems". Specific directives produce better results.

Keep Skills Focused

A skill for "react-components" should cover React component patterns only — not TypeScript, testing, or state management. Those warrant their own skills. Focused skills are easier to maintain and more composable.

Recommended Sections

  • When to use this skill (context/trigger)
  • Core patterns (with code examples)
  • Naming conventions
  • Error handling patterns
  • What to avoid (anti-patterns with explanations)
  • Project structure (if applicable)

Quality Standards

Skills are reviewed before merging. Here is what the review looks for:

Accurate frontmatter
All required fields present, category is valid, tags are relevant and lowercase.
Code examples work
All code examples are syntactically valid and follow the patterns described in the skill.
Clear and actionable
Instructions are specific and directive. Avoids vague guidance like "consider" or "might".
No overlap with existing skills
The skill covers a unique technology or a meaningfully different aspect not covered by existing skills.
Passes validation
Runs clean through npx tsx scripts/validate-skills.ts with no errors or warnings.
Expert-level content
Goes beyond beginner patterns. Covers edge cases, performance considerations, and production-ready approaches.

Submit a Pull Request

When your skill is ready, submit a pull request against the main branch of the Skillbox repository.

PR Title Format

Use the format: feat: add <skill-name> skill to <category> category

PR Description Checklist

Your PR description should confirm:

  • The skill passes npx tsx scripts/validate-skills.ts with no errors
  • The frontmatter is complete with accurate tags and category
  • All code examples in the skill are syntactically correct
  • The skill does not significantly overlap with an existing skill
  • You have personally used this technology in production

Review Process

Maintainers will review your PR within a few days. They may request changes to improve clarity, accuracy, or scope. Once approved, the skill is merged and immediately available to install.

First-time contributors: the repo has a CONTRIBUTING.md file with additional setup instructions and development workflow details.
